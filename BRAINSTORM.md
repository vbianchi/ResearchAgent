BRAINSTORM.md - ResearchAgent Project (v2.6.0 In Progress)
==========================================================

This document tracks current workflow ideas, user feedback, and immediate brainstorming for the ResearchAgent project. For longer-term plans and phased development, please see `ROADMAP.md`.

Current Version & Focus (v2.6.0 - LangGraph Core Integration & Direct Execution Paths):

The ResearchAgent is actively developing its LangGraph-based backend.

-   Key Backend Advancements (LangGraph Integration):

    -   Server Integration Complete: The LangGraph agent is now running within the main application server (`server.py`, `agent_flow_handlers.py`).

    -   `DIRECT_QA` Path Operational: Simple questions not requiring tools are handled by a `DirectQANode` in the graph, with `OverallEvaluatorNode` presenting the LLM's answer.

    -   `DIRECT_TOOL_REQUEST` Path Operational: Explicit user requests to use specific tools (e.g., "web search for X", "read file Y") are now handled by a `DirectToolExecutorNode` within the graph. This node dynamically loads and executes the requested tool, with the `OverallEvaluatorNode` formatting and presenting the output. This path uses the existing dynamic tool loading mechanism.

    -   State Management & Callbacks: The `ResearchAgentState` and `WebSocketCallbackHandler` are functioning correctly for these direct execution paths, providing real-time UI updates.

    -   Pydantic v2 Migration: Most data models, including tool inputs/outputs and LangGraph state, are now using Pydantic v2.

-   Immediate Backend Focus: Full Plan Execution in LangGraph:

    -   The top priority is now to implement the complete Plan-Code-Execute-Evaluate (PCEE) workflow within the LangGraph architecture. This involves replacing the current placeholder nodes for `planner`, `controller`, `executor`, and `step_evaluator` with their full logic.

    -   This will leverage the previously developed (and tested in isolation) concepts for multi-step plan generation, tool selection and input formulation by the controller, robust tool/LLM execution, and step-by-step evaluation with retry capabilities.

UI/UX Feedback & Ideas (To be revisited/prioritized post-full-LangGraph-PCEE integration):

1.  Plan Confirmation UI (High Priority - Depends on PlannerNode Output):

    -   Issue: The UI needs a clear way to display plans generated by the `PlannerNode` and allow user confirmation before execution by the LangGraph.

    -   Note: The `display_plan_for_confirmation` message type (currently triggered by the legacy planner logic in `agent_flow_handlers.py`) will need to be adapted or replaced to work with plans streamed or outputted by the LangGraph `PlannerNode`.

2.  Chat Clutter & Plan Display Format (High Priority - Post LangGraph Integration):

    -   User Feedback: Desire for a cleaner interface, distinguishing agent-user messages from status updates.

    -   LangGraph Implication: Event streaming from LangGraph nodes should allow better control over message types sent to chat vs. monitor.

3.  Intent Classification for "List Tools" (Ongoing Refinement):

    -   Issue: Queries like "What tools do you have?" are sometimes misclassified as `DIRECT_QA`.

    -   Goal: Such meta-queries should ideally be classified as `PLAN` so that the `PlannerNode` can generate a step to list tools, and the `ExecutorNode` can then use an LLM (with the `tools_summary`) to answer. Prompt engineering for `intent_classifier.py` is addressing this.

Illustrative Workflow (Conceptual - `DIRECT_TOOL_REQUEST` Path - Now Functional):

1.  User: "Read the file `report_findings.txt` from my workspace."

2.  `agent_flow_handlers.py` calls `intent_classifier.py`:

    -   Classifies as `DIRECT_TOOL_REQUEST`.

    -   Identifies `tool_name: "read_file"`.

    -   Extracts `extracted_tool_input: "report_findings.txt"`.

3.  `agent_flow_handlers.py` prepares initial `ResearchAgentState` and invokes `research_agent_graph.astream_events(...)`.

4.  LangGraph Execution:

    -   `intent_classifier_node` (graph node): Receives intent, tool name, input from state.

    -   Graph routes to `direct_tool_executor_node`.

    -   `direct_tool_executor_node`:

        -   Loads available tools for the current task.

        -   Selects `read_file` tool.

        -   Executes `read_file.arun(tool_input="report_findings.txt", config=...)`. (Callbacks for tool start/end are triggered).

        -   Places the file content (or error) into `state.executor_output`.

    -   Graph routes to `overall_evaluator_node`.

    -   `overall_evaluator_node`:

        -   Receives file content via `state.executor_output`.

        -   Uses a specific prompt for `read_file` output to tell its LLM to present the content directly (truncating if very long).

        -   The LLM call triggers `WebSocketCallbackHandler` (`on_llm_end`), which sends the formatted file content as an `agent_message` to the UI.

5.  Graph execution ends.

This document will be updated as we progress with the full PCEE LangGraph implementation.
